Malware Classes & Key Features
virus · worm · trojan · ransomware · spyware · RAT · adware · botnet

Malware Taxonomy

Virus
Attaches itself to legitimate executable files. Requires user action to spread — someone must execute the infected file. Replicates by infecting other files on the same system.
⟶ Modified file timestamps on executables ⟶ Unusual file size changes to .exe/.dll ⟶ AV alerts on known-good binaries ⟶ Degraded system performance
Worm
Self-replicating — spreads across networks without user action by exploiting vulnerabilities in network services. Can cause network saturation through mass self-propagation.
⟶ Sudden high outbound network traffic ⟶ Mass connection attempts to other hosts ⟶ SMB/SSH/RDP scan activity ⟶ Identical process spawning repeatedly
Trojan
Disguised as legitimate/useful software. No self-replication — user installs it willingly. Payload is typically a backdoor, RAT, or downloader. Named after the Trojan horse.
⟶ Unexpected network connections after install ⟶ New listening ports post-install ⟶ Process spawned from unexpected parent ⟶ Registry persistence keys added
Ransomware
Encrypts victim files and demands payment for the decryption key. Modern variants use double extortion: steal data first, then encrypt. Payment typically in cryptocurrency.
⟶ Mass file rename/extension changes ⟶ vssadmin delete shadows (VSS deletion) ⟶ bcdedit /set recoveryenabled No ⟶ High CPU/disk I/O during encryption ⟶ Ransom note files dropped everywhere
Spyware / Keylogger
Silently monitors and records user activity — keystrokes, screenshots, clipboard, credentials — and exfiltrates data to attacker. Often bundled with freeware or dropped by trojans.
⟶ Unexpected outbound HTTPS/DNS traffic ⟶ Hook on keyboard/mouse input APIs ⟶ Hidden processes with small memory footprint ⟶ Encrypted files in temp directories
RAT (Remote Access Trojan)
Gives attacker full remote control of the compromised host — shell access, file management, webcam/mic access, screenshot capture. Usually communicates via encrypted C2 channel.
⟶ Persistent outbound connection to C2 ⟶ Beaconing traffic (regular intervals) ⟶ New user account creation ⟶ RDP/VNC enabled unexpectedly
Rootkit
Provides persistent privileged access while hiding its presence. Patches kernel or user-space to conceal files, processes, registry keys, and network connections from security tools.
⟶ Discrepancy between tool output and raw disk ⟶ SSDT/IAT/DKOM hooks detected ⟶ Hidden processes visible only via raw memory ⟶ Kernel module not in lsmod but in memory
Botnet Agent
Infected host becomes a "bot" controlled by a botmaster via C2 infrastructure. Used for: DDoS, spam campaigns, credential stuffing, cryptocurrency mining, lateral movement.
⟶ Regular beaconing to hardcoded C2 IPs/domains ⟶ DGA (domain generation algorithm) DNS lookups ⟶ Unusual volume of outbound traffic ⟶ IRC/HTTP/HTTPS C2 channel traffic
Adware
Displays unwanted advertisements, often by injecting ads into browser sessions or replacing legitimate ads. Usually not destructive but degrades performance and may track browsing.
⟶ Browser extension modifications ⟶ Homepage/search engine changed ⟶ Injected ad network requests in traffic ⟶ New startup entries in registry

Malware Delivery Methods

Initial Access Vectors
  • Phishing — malicious email attachments (macros in Office docs, LNK files, ISO/ZIP archives)
  • Spear phishing — targeted phishing with personalized lures using OSINT
  • Drive-by download — visiting a compromised website triggers exploit via browser vulnerability
  • Malvertising — malicious ads on legitimate sites deliver exploits
  • Supply chain — compromising software update mechanisms or build pipelines (SolarWinds)
  • USB / physical media — autorun or social engineering to execute from removable media
  • Exploit public-facing app — unpatched web server, VPN, or RDP vulnerability
Persistence Mechanisms
  • Registry Run keys — HKLM/HKCU\...\CurrentVersion\Run
  • Scheduled tasks — schtasks (Windows), cron (Linux)
  • Services — malware installs itself as a Windows service or systemd unit
  • Startup folders — %APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup
  • DLL hijacking — plant malicious DLL in path searched before legitimate one
  • Shell init files — ~/.bashrc, ~/.profile, /etc/rc.local injection
  • Boot sector — MBR/UEFI infection for pre-OS persistence (bootkit)

Rootkit Types
kernel · user-mode · bootkit · hypervisor · infection · privilege levels · detection

Rootkit Classification by Privilege Level

Kernel-Mode Rootkit (Ring 0)

Operates at the highest privilege level — inside the OS kernel. Can modify kernel data structures directly: the SSDT (System Service Descriptor Table), IDT (Interrupt Descriptor Table), process/module lists. All OS tools rely on the kernel for information — the kernel lies to them.

  • Infection method: Exploits kernel vulnerability, malicious driver (BYOVD), or disabling Driver Signature Enforcement
  • Privilege level: Ring 0 — same as OS kernel
  • Detection: Cross-view comparison (live kernel vs raw disk/memory), integrity checkers (chkrootkit, rkhunter), memory forensics (Volatility)
  • Hides: Files, processes, network connections, registry keys, loaded modules
  • Example: TDL4, Necurs, ZeroAccess
User-Mode Rootkit (Ring 3)

Operates in user space. Cannot modify the kernel, so hooks user-space libraries instead — patches DLLs (Windows) or LD_PRELOAD shared objects (Linux) to intercept and filter API calls before they return to the calling program.

  • Infection method: Trojan, exploit granting user-level execution
  • Privilege level: Ring 3 — runs as an unprivileged or admin user
  • Detection: Easier than kernel rootkits — kernel is trustworthy; compare API results to raw system calls directly
  • Hides: Files and processes from the user who owns it; weaker hiding than kernel rootkit
  • Example: Hacker Defender, various Linux LD_PRELOAD rootkits
Bootkit (MBR / UEFI)

Infects the Master Boot Record or UEFI firmware — loads before the operating system. Establishes control before any OS security mechanisms activate. Survives OS reinstallation (disk wipe required; UEFI variants may survive even that).

  • Infection method: Physical access, kernel exploit that writes to MBR, compromised firmware update
  • Privilege level: Pre-Ring 0 — runs before kernel exists
  • Detection: Compare MBR bytes to known-good; offline boot scanning; UEFI Secure Boot prevents unsigned bootloaders
  • Hides: Entire malware stack — loaded before any AV or security tool
  • Example: Stoned Bootkit, Rovnix, LoJax (UEFI), TDL4 (MBR)
Hypervisor Rootkit (Ring -1)

Moves the legitimate OS into a virtual machine, placing the rootkit beneath the OS as a thin hypervisor layer. The OS has no awareness it's been virtualized. Extremely difficult to detect from within the compromised OS.

  • Infection method: Kernel-level compromise then installs hypervisor; or direct hardware access
  • Privilege level: Ring -1 (VMX root mode) — more privileged than Ring 0
  • Detection: Timing anomalies (RDTSC instruction), CPUID hypervisor bit check, hardware-based monitoring (Intel TXT)
  • Hides: Everything — from underneath the entire OS
  • Example: Blue Pill (proof-of-concept), SubVirt (Microsoft Research)

Indicators of Compromise (IOCs) — Categories

// What Are IOCs?
Indicators of Compromise are artifacts observed on a host or network that indicate a system has been compromised. They are split into host-based (what's on the machine) and network-based (what's on the wire). IOCs can be shared between organizations via STIX/TAXII formats and threat intelligence platforms.

Botnets & Command and Control
C2 architecture · DGA · protocols · takedown · impact

Botnet Architecture

Key Components
Botmaster
The attacker who controls the botnet. Issues commands through the C2 infrastructure.
C2 Server
Command and Control server — receives check-ins from bots, issues commands, collects exfiltrated data.
Bot
An infected host running the bot agent. May be a PC, server, IoT device, or router. Receives commands and carries them out.
Botnet
The collective of all infected bots under a single botmaster's control. Ranges from hundreds to millions of devices.

C2 Communication Models

Centralized C2
  • All bots connect to one (or a small cluster of) C2 servers
  • Simple to operate; fast command propagation
  • Weakness: Takedown or sinkholing of the C2 server kills the entire botnet
  • Common protocols: HTTP/HTTPS (blends with normal traffic), IRC (legacy), custom TCP
  • Detection: netstat shows persistent connection to one external IP
Peer-to-Peer (P2P) C2
  • Bots communicate with each other; no single C2 server
  • Resilient: Removing one node doesn't kill the botnet
  • Commands propagate through the peer network
  • Much harder to take down — requires poisoning or sinkholing the peer list
  • Example: Storm Worm, Zeus P2P variant, Necurs

Domain Generation Algorithms (DGA)

Evading C2 Takedown with DGA

A DGA generates hundreds or thousands of pseudo-random domain names per day (using a seed like the current date). The botmaster only needs to register one — the bot tries all generated domains and connects to whichever one resolves. Defenders cannot easily block all generated domains in advance.

# Simplified DGA example (seed = date) # For date 2024-01-15, generate domains: xkzqmfvrlpwa.com # bot tries this... bqtynvshoiez.com # and this... wjoamdkpxcqb.net # and this... (one is registered = C2) # Detection: high NXDOMAIN rate + high-entropy domain names # Mitigation: DGA domain prediction + preemptive sinkholing

Live Response Triage Sequence

// Order of Volatility
Collect volatile evidence first — it disappears when the system is powered off. CPU registers → running processes/memory → network connections → open files → disk data → backup/archive media.

Network Monitoring Tools & Techniques
Wireshark · tcpdump · Snort · Zeek · SIEM · NetFlow

Packet Capture Tools

Wireshark
  • GUI-based packet analyzer — capture and dissect network traffic in real time
  • Supports 3000+ protocol dissectors — automatically parses HTTP, DNS, TLS, SMB, etc.
  • Display filters: ip.addr == 10.0.0.1, tcp.port == 443, http.request
  • Follow TCP/UDP stream to reconstruct full conversations
  • Reads pcap/pcapng files — standard format for captured traffic
  • Coloring rules highlight anomalies (RST storms, malformed packets)
tcpdump
  • CLI packet capture — lightweight, runs on almost any Unix system
  • Writes pcap files readable by Wireshark
  • Common syntax: tcpdump -i eth0 -w capture.pcap
  • Filtering: tcpdump host 8.8.8.8 and port 53
  • Show ASCII content: tcpdump -A -s0 port 80
  • Essential on servers without GUI — the field standard for remote capture

Intrusion Detection / Prevention Systems

Snort (IDS/IPS)
  • Open-source network IDS/IPS — the most widely deployed in the world
  • Rule-based detection: rules match packet content, headers, payloads
  • Three modes: Sniffer (print packets), Packet Logger (save to disk), NIDS (apply rules and alert)
  • Rule format: alert tcp any any -> $HOME_NET 22 (msg:"SSH scan"; flags:S; threshold:type both, track by_src, count 5, seconds 60; sid:1000001;)
  • Maintained community ruleset + commercial VRT ruleset
  • As IPS: inline mode drops matching packets in real time
Zeek (formerly Bro)
  • Network analysis framework — not rule-based but generates structured logs
  • Produces high-fidelity logs: conn.log, dns.log, http.log, ssl.log, files.log
  • Ideal for behavioral analysis and threat hunting — not just alerting
  • Extracts file hashes, X.509 certificates, software versions from traffic
  • Scriptable in the Zeek scripting language for custom detections
  • Widely used in large-scale network security monitoring (NSM)

Flow-Based Monitoring

NetFlow / IPFIX / sFlow

Instead of capturing full packets (expensive at scale), flow monitoring records metadata about each conversation — source/destination IP, port, protocol, byte/packet counts, start/end time. Routers and switches export flow records to a collector. Useful for: detecting anomalous traffic volumes, tracking lateral movement, identifying heavy talkers, investigating incidents when full pcap isn't available.

NetFlow v5/v9
Cisco standard. Widely supported. Router exports 5-tuple + counters.
IPFIX
IETF standard (RFC 7011). NetFlow v9 successor. Template-based, extensible.
sFlow
Statistical sampling — samples 1-in-N packets. Less precise but lower overhead at high speed links.

SIEM (Security Information and Event Management)

What a SIEM Does

A SIEM aggregates logs and events from across the environment — firewalls, IDS, endpoints, servers, authentication systems — correlates them, and generates alerts when event combinations match known attack patterns. Core capabilities:

  • Log aggregation: Collects from Windows Event Log, syslog, cloud APIs, endpoint agents
  • Normalization: Converts different log formats to a common schema
  • Correlation: Links related events across sources — failed login + new service + outbound connection = alert
  • Alerting: Triggers on rule matches or statistical anomalies
  • Search/Investigation: Analysts query historical logs across all sources from one interface
  • Retention: Long-term log storage for compliance and forensic investigation
  • Examples: Splunk, IBM QRadar, Microsoft Sentinel, Elastic SIEM, ArcSight

Nmap — Network Reconnaissance Tool

Nmap Essentials

Nmap (Network Mapper) is the standard tool for network discovery and security auditing. Used by defenders (asset inventory, vulnerability assessment) and attackers (reconnaissance).

nmap 192.168.1.0/24 # Ping sweep — which hosts are up? nmap -sS 192.168.1.5 # SYN (stealth) scan — half-open nmap -sV -sC 192.168.1.5 # Service version + default scripts nmap -O 192.168.1.5 # OS fingerprinting nmap -p 1-65535 192.168.1.5 # All ports nmap -sU -p 53,161,123 target # UDP scan specific ports nmap -A target # Aggressive: OS + version + scripts + traceroute

CVE & CVSS Scoring

CVE (Common Vulnerabilities & Exposures)
  • Standardized identifier for publicly known vulnerabilities
  • Format: CVE-YEAR-NUMBER (e.g., CVE-2021-44228 = Log4Shell)
  • Managed by MITRE; published in the National Vulnerability Database (NVD)
  • Every CVE has a description, affected products, and often a CVSS score
  • Vendors reference CVE IDs in security advisories and patch notes
CVSS (Common Vulnerability Scoring System)
  • Numerical score 0.0–10.0 quantifying vulnerability severity
  • Critical: 9.0–10.0  |  High: 7.0–8.9  |  Medium: 4.0–6.9  |  Low: 0.1–3.9
  • Base metrics: Attack Vector, Attack Complexity, Privileges Required, User Interaction, Scope, Confidentiality/Integrity/Availability Impact
  • Current version: CVSS v3.1
  • Not the only factor in patching priority — exploitability in the wild matters more

Amplification / Reflection DDoS

How Amplification Attacks Work

Attacker sends a small request with the victim's IP as the spoofed source to many open reflectors (DNS servers, NTP servers, memcached). Each reflector sends a much larger response to the victim. The amplification factor (response/request size ratio) determines how effective the attack is.

ProtocolPortAmplification Factor
DNS ANY queryUDP/5328–54×
NTP monlistUDP/123Up to 556×
MemcachedUDP/11211Up to 51,000×
SSDP (UPnP)UDP/190030×

Defense: BCP38 — ISPs should filter spoofed source IPs at their border. Rate-limiting DNS responses. Disabling NTP monlist (ntp.conf: restrict default nomonlist).

Windows Security Mechanisms Reference

MechanismWhat It DoesBypass / Limitation
UAC (User Account Control)Even admin accounts run with limited tokens by default. Applications must request elevation — triggers consent or credential prompt. Reduces silent privilege escalation.UAC bypass techniques (fodhelper, eventvwr, sdclt hijacks) elevate without prompt. Disabled on some enterprise builds.
Windows Defender AVBuilt-in antivirus: signature + cloud + behavioral detection. Integrates with Windows Security Center. Always-on real-time protection.Can be disabled by SYSTEM-level processes. Bypassed by obfuscation, AMSI bypass scripts, living-off-the-land techniques.
Windows FirewallHost-based stateful firewall. Inbound/outbound rules per profile (Domain, Private, Public). Blocks unsolicited inbound by default.Malware with admin access can add rules or disable it. Outbound is often permissive by default.
AppLockerApplication whitelisting — allow/deny execution of .exe, .dll, .msi, .ps1, .vbs by path, publisher signature, or hash. Significantly raises bar for malware execution.LOLBAS (regsvr32, mshta, wmic, rundll32) bypass path rules. Requires Enterprise/Education SKU.
WDAC (Windows Defender Application Control)More comprehensive than AppLocker — enforced at kernel level via code integrity policy. Blocks unsigned code even for admins.Policy bypass requires signed trusted binaries or policy modification (needs admin). More robust than AppLocker.
Credential GuardUses Hyper-V virtualization to isolate LSASS credentials in a separate Secure World VM — Mimikatz cannot access the credential material even with SYSTEM access.Requires UEFI, Secure Boot, and compatible hardware. Does not protect against all credential attacks (e.g., pass-the-hash still works against remote services).
DEP / NXMarks stack, heap, and data pages as non-executable — shellcode injected into those regions cannot run. CPU hardware-enforced.Bypassed by ROP (Return-Oriented Programming) — executes existing code, not injected shellcode.
ASLRRandomizes base addresses of stack, heap, and loaded modules on each execution — makes exploiting buffer overflows harder (addresses unpredictable).Information disclosure vulnerabilities can leak addresses. Low-entropy on 32-bit systems allows brute force.
BitLockerFull-volume encryption for Windows drives. Protects data at rest — drive removed from machine = unreadable without key. TPM-backed key storage.Protects against offline attacks only. No protection against live, authenticated attacks on a running system.

Log forwarding to SIEM reduces this risk.

Protected Users Security Group

Hardening Privileged Accounts

Adding an account to the Protected Users security group (Windows Server 2012 R2+) applies additional restrictions: NTLM authentication is disabled for that account (Kerberos only), credentials are not cached on the workstation, DES/RC4 Kerberos encryption is disabled (AES required), Kerberos TGTs expire after 4 hours (not renewable to 10 hours). Significantly hardens against credential theft attacks.

Built-in Unix/Linux Security Mechanisms
DAC · MAC · SELinux · AppArmor · seccomp · iptables · sudo · PAM

Linux Security Mechanisms Reference

MechanismWhat It DoesNotes
DAC (Discretionary Access Control)Standard Unix permission model — owner, group, other with read/write/execute bits. Owner controls access to their files. chmod / chown.Basis of all Unix security. Supplemented by ACLs (setfacl/getfacl) for finer control.
MAC — SELinuxMandatory Access Control. NSA-developed. Type enforcement: every process and file has a security context. Policy defines what types can access what. Even root is confined by policy.Default on RHEL/CentOS/Fedora. Modes: Enforcing (blocks), Permissive (logs only), Disabled. Commands: getenforce, setenforce, sestatus, audit2allow.
MAC — AppArmorPath-based MAC. Profiles define which files, capabilities, and network operations a process can use. Simpler than SELinux to configure.Default on Ubuntu/Debian. Modes: Enforce, Complain. Commands: aa-status, aa-enforce, apparmor_parser.
seccompSyscall filter — restricts which system calls a process can make. Reduces kernel attack surface dramatically. Process cannot escalate even if code execution achieved.Used by Docker, Chrome, OpenSSH. Seccomp-BPF allows fine-grained per-syscall rules with BPF programs.
sudoAllows specific users to execute specific commands as root (or another user). Configured in /etc/sudoers. Logs all usage. Avoids sharing the root password.Edit with visudo only. Dangerous config: NOPASSWD: ALL. Check for misconfigs: sudo -l
PAM (Pluggable Authentication Modules)Modular authentication framework. Configures authentication, authorization, session setup, and password management. Allows MFA, LDAP auth, lockout policies.Configuration: /etc/pam.d/. Modules: pam_unix (local), pam_ldap, pam_google_authenticator (MFA), pam_tally2 (lockout).
iptables / nftablesLinux kernel packet filtering — host-based firewall. Rules in INPUT/OUTPUT/FORWARD chains. Stateful tracking via conntrack.iptables = legacy (still widely used). nftables = modern replacement. ufw (Ubuntu) and firewalld (RHEL) are management layers on top.
SSH key authenticationPasswordless authentication using RSA/ECDSA/Ed25519 key pairs. Private key stays on client; public key on server in ~/.ssh/authorized_keys. Immune to password brute-force.Disable password auth: PasswordAuthentication no in sshd_config. Use AllowUsers/AllowGroups to restrict who can SSH.
CapabilitiesDivides root privileges into distinct units (capabilities). Process can be granted only the capabilities it needs (e.g., CAP_NET_BIND_SERVICE) without full root.getcap, setcap commands. Principle of least privilege — avoids running services as full root. Misconfigured capabilities = privilege escalation path.
chroot / namespaces / cgroupschroot: restricts filesystem view. Namespaces: isolate PID, network, mount, user spaces. cgroups: limit resource usage. Together = basis of Linux containers.chroot alone is weak (root can escape). Namespaces + seccomp + capabilities = container security model.
