Encapsulation: How Data Travels Down the Stack

As data moves from application to wire, each layer adds a header (and sometimes a trailer), wrapping the previous layer's data. This is called encapsulation. At the receiver, each layer strips its header — de-encapsulation.

IPv6 Basics

IPv6 Format
  • 128-bit addresses written as 8 groups of 4 hex digits: 2001:0db8:85a3:0000:0000:8a2e:0370:7334
  • Leading zeros in each group can be omitted: 2001:db8:85a3:0:0:8a2e:370:7334
  • :: replaces one consecutive sequence of zero groups: 2001:db8::8a2e:370:7334
  • ::1 = loopback (IPv4 equivalent of 127.0.0.1)
  • fe80::/10 = link-local addresses (auto-configured, not routed)
  • 2000::/3 = global unicast (publicly routable)
  • Standard subnet prefix: /64
IPv6 Address Types
Unicast
One-to-one delivery. Global (2000::/3), link-local (fe80::/10), loopback (::1)
Multicast
ff00::/8 — one-to-many. Replaces broadcast. ff02::1=all nodes, ff02::2=all routers
Anycast
One-to-nearest — same address on multiple nodes; routing sends to geographically nearest
No Broadcast
IPv6 has no broadcast — multicast is used instead. This improves efficiency.

Ethernet Frame Structure

# IEEE 802.3 Ethernet Frame (standard) [ Preamble 7B ] [ SFD 1B ] [ Dst MAC 6B ] [ Src MAC 6B ] [ EtherType 2B ] [ Payload 46-1500B ] [ FCS 4B ] # Preamble: 7 bytes of 10101010 for clock sync # SFD (Start Frame Delimiter): 10101011 signals frame start # EtherType common values: 0x0800 = IPv4 0x0806 = ARP 0x86DD = IPv6 0x8100 = 802.1Q VLAN tag # FCS: CRC-32 checksum for error detection (not correction) # MTU (Maximum Transmission Unit) = 1500 bytes payload # Jumbo frames = up to 9000 bytes (supported by most enterprise NICs/switches)

MAC Addresses

Structure of a MAC Address

MAC (Media Access Control) addresses are 48-bit (6-byte) identifiers assigned to NICs. Written as six pairs of hex octets: 00:1A:2B:3C:4D:5E.

# First 3 bytes = OUI (Organizationally Unique Identifier) — identifies manufacturer 00:50:56 = VMware 08:00:27 = VirtualBox 00:0C:29 = VMware Workstation 00:50:B6 = Buffalo FC:FB:FB = Cisco # Last 3 bytes = NIC-specific (assigned by manufacturer) # Bit 1 of first byte: 0=unicast 1=multicast # Bit 2 of first byte: 0=globally unique 1=locally administered FF:FF:FF:FF:FF:FF = Broadcast (all devices on segment receive)

ARP — Address Resolution Protocol

IP → MAC Resolution

When a host knows the destination IP but needs the destination MAC (required to build an Ethernet frame), it uses ARP. ARP is a Layer 2/3 boundary protocol — it's encapsulated directly in Ethernet frames (EtherType 0x0806), not in IP packets.

# ARP Exchange: Host A broadcasts: "Who has 192.168.1.1? Tell 192.168.1.50" Dst MAC = FF:FF:FF:FF:FF:FF (broadcast) Host B replies: "192.168.1.1 is at 00:11:22:33:44:55" Dst MAC = Host A's MAC (unicast) Host A caches: 192.168.1.1 → 00:11:22:33:44:55 in ARP table arp -a # View ARP cache (both Windows and Linux) # ARP spoofing: attacker sends unsolicited ARP replies # Poisoning victim's ARP cache to intercept traffic (MITM)

VLANs & 802.1Q Trunking

VLANs

Virtual LANs logically segment a physical switch into multiple broadcast domains. Devices in different VLANs cannot communicate at Layer 2 — inter-VLAN routing requires a Layer 3 device (router or Layer 3 switch). Each VLAN is identified by a number (1–4094). VLAN 1 is the default/native VLAN on most Cisco equipment.

802.1Q VLAN Tagging

Trunk ports carry frames from multiple VLANs. IEEE 802.1Q inserts a 4-byte tag between the Source MAC and EtherType fields: TPID (0x8100) + TCI (3-bit PCP priority + 1-bit DEI + 12-bit VLAN ID). Access ports connect end devices and strip/add tags automatically. The native VLAN is untagged on trunks.

Spanning Tree Protocol (STP)

Preventing Broadcast Storms

Without STP, redundant switch paths create Layer 2 loops — Ethernet frames have no TTL, so they loop forever, saturating the network. STP (IEEE 802.1D) elects a Root Bridge (lowest Bridge ID = priority + MAC) and blocks redundant ports to create a loop-free tree topology. RSTP (802.1w) provides faster convergence. PVST+ (Cisco) runs per-VLAN. STP port states: Blocking → Listening → Learning → Forwarding.

How Routers Make Forwarding Decisions

Longest Prefix Match

When a router receives a packet, it looks up the destination IP in its routing table and finds the most specific (longest prefix) match. If multiple routes match, the longest prefix wins. If the prefix lengths tie, Administrative Distance (AD) is used to prefer one routing source over another. If AD ties, metric decides.

# Routing table example — destination 10.1.1.5: 10.0.0.0/8 via 192.168.1.1 ← matches (/8) 10.1.0.0/16 via 192.168.1.2 ← matches (/16) 10.1.1.0/24 via 192.168.1.3 ← matches (/24) ← WINS (most specific) 0.0.0.0/0 via 192.168.1.254 ← default route (if no other match)

The practice test asks which protocol has AD 120 — answer is RIP. OSPF=110, EIGRP=90, BGP=200. Lower AD = more trusted. Connected (0) always beats static (1) beats any dynamic protocol.

Routing Protocol Types

Distance Vector (RIP, EIGRP)
  • Each router knows only hop count to destinations and which neighbor to forward to
  • Share routing tables with directly connected neighbors periodically
  • RIP: max 15 hops (16 = unreachable), updates every 30s, slow convergence, split horizon to prevent loops
  • EIGRP: Cisco; uses bandwidth + delay metric; DUAL algorithm prevents loops; faster than RIP
  • Susceptible to count-to-infinity problem; routing loops possible
Link State (OSPF, IS-IS)
  • Each router builds a complete map of the network (Link State Database = LSDB)
  • Routers flood LSAs (Link State Advertisements) to all routers in the area
  • Each router independently runs Dijkstra's SPF algorithm to compute best paths
  • OSPF: areas (Area 0 = backbone), DR/BDR election on multi-access networks, fast convergence
  • More CPU/memory intensive but scales better, no hop-count limit

NAT (Network Address Translation)

How NAT Works

NAT allows multiple private IP addresses to share one or more public IPs. The router/firewall maintains a NAT translation table mapping internal (private IP + port) to external (public IP + port). PAT (Port Address Translation) / "NAT overload" is the common form — many inside hosts share a single public IP using unique source port numbers. NAT modifies both IP and TCP/UDP headers (and recalculates checksums) as packets transit.

# Inside → Outside translation (PAT) 192.168.1.5:54321 → 8.8.8.8:53 # Original (DNS query) 203.0.113.1:1024 → 8.8.8.8:53 # After NAT (src IP+port translated) # NAT table entry: 192.168.1.5:54321 ↔ 203.0.113.1:1024 # Return traffic uses table to translate back

LAN vs WAN

LAN (Local Area Network)
  • Geographically limited — single building or campus
  • High speed: 100 Mbps – 400 Gbps Ethernet
  • Low latency: sub-millisecond
  • Technologies: Ethernet (802.3), Wi-Fi (802.11), VLANs
  • Organization owns/controls the infrastructure
  • VLAN: logical segmentation of a single physical LAN
  • MAN (Metropolitan Area Network): city-wide, between LANs
WAN (Wide Area Network)
  • Geographically distributed — cities, countries, global
  • Lower speed and higher latency than LAN
  • Technologies: MPLS, leased lines, SD-WAN, Frame Relay (legacy), DSL, fiber
  • Usually provided by a carrier/ISP — organization pays for service
  • The internet is the world's largest WAN
  • VPN overlays provide encrypted WAN connectivity over public internet

Network Security Zones

DMZ (Demilitarized Zone) Design

A DMZ is a network segment between the internet and the internal network, hosting public-facing services (web servers, mail servers, DNS). A firewall with three interfaces implements it: one leg to internet, one to DMZ, one to internal LAN. Traffic rules: internet can reach DMZ; DMZ cannot initiate connections to internal LAN; internal LAN can reach DMZ and internet.

Internet ──[Firewall]──┬── DMZ (web/mail/DNS servers) └── Internal LAN (workstations, DB servers) # If web server in DMZ is compromised, attacker still # cannot directly reach internal database servers

Stateful vs Stateless Firewalls

Stateless Firewall (ACL-based)

Evaluates each packet independently based on source/destination IP, port, and protocol. No memory of connection context. Must create explicit rules for both directions of traffic. Simple and fast. Cannot detect connection-level attacks (e.g., malformed TCP sequences).

Stateful Firewall

Maintains a connection state table. Tracks TCP three-way handshake, sequence numbers, and connection state. Automatically allows return traffic for established connections — no explicit inbound rule needed. Can detect TCP session hijacking, SYN floods. Standard for enterprise perimeter firewalls.

DNS Attacks

DNS Cache Poisoning

Attacker injects false DNS records into a resolver's cache. Future queries for that domain return the attacker's IP until TTL expires. Kaminsky attack (2008) showed how to efficiently poison caches. Defense: DNSSEC (cryptographically signed records), randomized source ports and transaction IDs.

DNS Tunneling

Data exfiltration or C2 communication encoded in DNS queries/responses. Attacker controls an authoritative NS; victim's queries for subdomains (encoded data as subdomain labels) reach the attacker's NS. Hard to detect — DNS is rarely blocked. Defense: DNS firewall, query rate limiting, detect long subdomain strings.

DHCP Process (DORA)

Discover → Client broadcasts: "I need an IP" (src=0.0.0.0 dst=255.255.255.255) Offer → Server unicasts/broadcasts: "Here's 192.168.1.50 for 24h" Request → Client broadcasts: "I'll take 192.168.1.50 from server X" Acknowledge → Server: "Confirmed, 192.168.1.50 is yours until [time]" # DHCP options also provide: subnet mask, default gateway, DNS servers, NTP # DHCP starvation attack: exhaust IP pool with spoofed MACs # Rogue DHCP: attacker's DHCP server wins race → assigns attacker's gateway/DNS # Defense: DHCP snooping on switches (only trust specific DHCP server ports)

Encryption Fundamentals
symmetric · asymmetric · PKI · TLS · key exchange · cipher suites

Symmetric Encryption

One Key for Both Operations

Both encryption and decryption use the same secret key. The key must be shared securely between parties beforehand (the "key distribution problem"). Fast — suitable for bulk data encryption.

AlgorithmKey SizeBlock SizeStatus
DES56-bit64-bitBroken — brute-forceable in hours
3DES112/168-bit64-bitDeprecated — Sweet32 attack, slow
AES-128/256128/256-bit128-bitCurrent standard. NIST approved. Use AES-256 for TS/SCI.
ChaCha20256-bitStream cipherModern. Faster than AES on non-hardware-accelerated CPUs. Used in TLS 1.3.
RC4VariableStream cipherBroken — statistical biases. Removed from TLS.

Cipher Modes of Operation

ECB
Electronic Codebook. Same plaintext → same ciphertext. Reveals patterns (famous ECB penguin). Never use.
CBC
Cipher Block Chaining. XOR with previous ciphertext block before encrypting. Requires IV. Sequential only.
CTR
Counter mode. Turns block cipher into stream cipher. Parallelizable. No padding needed.
GCM
Galois/Counter Mode. CTR + GHASH authentication. Provides authenticated encryption (AEAD). Standard in TLS 1.3.

Asymmetric Encryption

Public/Private Key Pairs

Two mathematically linked keys: public key (shared with everyone) and private key (kept secret). Operations: what one key encrypts, only the other can decrypt. Solves key distribution problem — share public key freely.

# Encryption use (confidentiality): Alice encrypts with Bob's PUBLIC key → only Bob's PRIVATE key can decrypt # Digital signature use (authentication + integrity): Alice signs with her PRIVATE key → anyone with her PUBLIC key can verify # RSA: security based on difficulty of factoring large primes # ECC: smaller keys, same security. 256-bit ECC ≈ 3072-bit RSA # ECDH/ECDHE: key agreement (shared secret without transmitting key)

TLS Handshake (TLS 1.3)

PKI (Public Key Infrastructure)

Certificate Authorities
  • Root CA: Self-signed cert. Trust anchor. Offline in secure facility. Signs Intermediate CAs.
  • Intermediate CA: Signs end-entity (server/client) certs. Online. If compromised, revoke without affecting root.
  • Certificate chain: Server cert → Intermediate CA → Root CA. Browser verifies full chain.
  • X.509: Standard certificate format. Contains: subject, issuer, public key, validity period, extensions, signature.
Perfect Forward Secrecy (PFS)

PFS ensures that compromise of a server's long-term private key does NOT allow decryption of past sessions. Achieved with ephemeral key exchange (ECDHE, DHE) — a new session key is generated per session and discarded after. TLS 1.3 mandates PFS. TLS 1.2 with RSA key exchange lacks PFS — one key compromise = all recorded sessions decryptable.

Kerberos & Active Directory
TGT · TGS · KDC · AD vs DC · authentication flow

Kerberos Authentication Flow

Step-by-Step Process
Step 1: Client requests a TGT from the KDC (Key Distribution Center) Step 2: KDC verifies credentials → sends back encrypted TGT + session key TGT is encrypted with the TGS secret key Step 3: Client stores TGT. When it expires, Session Manager auto-renews transparently Step 4: Client sends TGT + SPN (Service Principal Name) to the TGS Step 5: KDC verifies TGT and checks user has access to the requested service Step 6: TGS issues a service ticket (encrypted with that service's secret key) Step 7: Client presents service ticket to the target service → access granted
// Why TGTs exist
TGTs avoid repeatedly asking the user for their password (which Kerberos uses to derive a master key) or storing the master key on the workstation. The TGT proves identity once; service tickets are issued without re-prompting.

Kerberos Key Facts

  • Default authentication in Microsoft Windows since Windows Server 2000
  • Also implemented in: Apple macOS, FreeBSD, Linux (via MIT Kerberos / Heimdal)
  • Standard for websites and Single Sign-On (SSO) implementations
  • Uses tickets (TGT, service tickets) — not passwords — for ongoing authentication
  • Port: TCP/UDP 88
  • Provides mutual authentication — both client and server verify each other's identity
  • Messages are protected against eavesdropping and replay attacks

Active Directory vs Domain Controllers

Domain Controllers
  • Hold the key to Active Directory — they run and serve it
  • Exist in every domain (not every domain has Active Directory)
  • Windows implementations: Microsoft Active Directory, Microsoft Azure AD
  • Linux implementation: Samba (can act as AD-compatible DC)
  • A domain can have multiple DCs for redundancy and performance
Active Directory
  • A type of domain — not just a server
  • Does not exist in every network — only in Windows domains configured with it
  • Stores: user accounts, computer accounts, groups, policies, trusts
  • Provides: authentication (Kerberos/NTLM), authorization, Group Policy, DNS integration
  • The DC is the server; AD is the directory service the DC hosts

IGP vs EGP

IGP — Interior Gateway Protocol
  • Used internally within a single Autonomous System
  • Typically used by ISP core routers internally
  • Examples: OSPF, EIGRP, RIP, IS-IS
EGP — Exterior Gateway Protocol
  • Used between different Autonomous Systems (ISP to ISP)
  • Typically used by edge routers
  • Primary modern example: BGP (Border Gateway Protocol)

Routing Metric Comparison: RIP vs OSPF

RIP — Hop Count

RIP selects the path with the fewest router hops, regardless of link speed. A 3-hop path through 56K modems beats a 4-hop path through gigabit fiber. Maximum 15 hops; 16 = unreachable. Updates every 30 seconds.

OSPF — Cost (Bandwidth-Based)

OSPF calculates cost = reference bandwidth ÷ interface bandwidth. Faster links have lower cost. Default reference: 100 Mbps. FastEthernet=cost 1, T1=cost 64. OSPF selects the lowest total-cost path — correctly prefers faster links.

EIGRP Composite Metric

EIGRP Metric Components

EIGRP uses a composite metric based on: Bandwidth (most impactful) + Delay (cumulative path delay) by default. Additionally configurable: Reliability, Load, MTU — but these are disabled by default. The DUAL (Diffusing Update ALgorithm) guarantees loop-free paths and fast convergence by maintaining a Feasible Successor (backup route) pre-computed.

Adjacency

Router Adjacency

Routers that are connected to one another, know all routes, and work together to determine the best route are called adjacent. In OSPF, adjacency is the full state between two routers that have exchanged their full link-state databases. In EIGRP, neighbors become adjacent after exchanging Hello packets and agreeing on parameters. Adjacency is a prerequisite to exchanging routing updates.

Protocol Headers & SEQ/ACK Numbers
TCP · UDP · Ethernet · sequence numbers · acknowledgment

TCP Header

20 Bytes Minimum — Fields

Source Port (2B) | Destination Port (2B) | Sequence Number (4B) | ACK Number (4B) | Flags (2B) | Window Size (2B) | Checksum (2B) | Urgent PTR (2B) | Options (variable)

  • Total header: 20 bytes minimum, up to 60 bytes with options
  • Max TCP payload per Ethernet frame: 1460 bytes (1500 MTU − 20 IP − 20 TCP)

UDP Header

8 Bytes Fixed — Fields

Source Port (2B) | Destination Port (2B) | Length (2B) | Checksum (2B)

  • Total header: exactly 8 bytes, fixed — no options
  • Max UDP payload per Ethernet frame: 1472 bytes (1500 MTU − 20 IP − 8 UDP)

Ethernet Frame

Ethernet II Frame Structure

Preamble (7B) | SFD (1B) | Destination MAC (6B) | Source MAC (6B) | EtherType (2B) | Payload (46–1500B) | FCS/Checksum (4B)

  • EtherType 0x0800 = IPv4  |  0x0806 = ARP  |  0x86DD = IPv6  |  0x8100 = 802.1Q
  • Minimum frame size: 64 bytes — short payloads padded to 46 bytes minimum
  • Maximum payload (MTU): 1500 bytes

TCP Sequence & ACK Numbers

Sequence Numbers
  • 32 bits — used by the transmitter on each transmitted packet
  • SEQ numbers are randomly chosen when a host initiates a TCP session (ISN — Initial Sequence Number)
  • Track the byte offset of data being sent
  • Enable in-order reassembly at the receiver
ACK Numbers
  • Inform the sender that transmitted data was received successfully
  • ACK number = SEQ number + amount of data received: SEQ + DATA = ACK
  • Means "I received everything up to ACK−1, send me starting at ACK"
  • Example: SEQ=1000, 400 bytes sent → receiver ACKs with ACK=1400

NETSTAT

What NETSTAT Provides

NETSTAT provides information and statistics about protocols in use and current TCP/IP network connections. Shows active connections with state (ESTABLISHED, LISTEN, TIME_WAIT), local and foreign addresses:ports, and — with -p — the owning process/PID. Essential for identifying unexpected listeners and active sessions during incident response.

netstat -an # All connections, numeric (Windows & Linux) netstat -tulpn # Linux: TCP/UDP listening with PID netstat -b # Windows: show owning executable

HTTPS: Hybrid Encryption Flow

How HTTPS Uses Both Asymmetric and Symmetric Encryption

A HTTPS connection uses both symmetric and asymmetric encryption. Asymmetric is used first to establish trust and exchange a secret; symmetric is used for all bulk data transfer.

Asymmetric Phase

  1. Client connects → server sends its TLS certificate. CA verifies the certificate and server identity.
  2. Certificate also contains cipher suites and maximum TLS version supported.
  3. After certificate verification and algorithm negotiation, client generates a session key. Session key is encrypted with the server's public key and sent.
  4. Server decrypts the session key using its private key.

Symmetric Phase

  1. The session key is used for symmetric encryption of all subsequent data exchange.
  2. Session continues with only symmetric encryption — asymmetric was only for key exchange.
// Session Keys
Session keys are randomly generated per session and discarded afterward. This is what enables Perfect Forward Secrecy — even if the server's private key is later compromised, past session keys (and thus past traffic) cannot be recovered.

PGP — Pretty Good Privacy

PGP Encryption Model
  • Used for email and file encryption
  • Uses a Web of Trust model instead of a central CA — users sign each other's public keys
  • Combines both encryption types: symmetric for data (fast), asymmetric for key exchange (secure)
  • Sender encrypts data with a random session key → encrypts session key with recipient's public key → recipient uses private key to decrypt session key → decrypts data
  • Signing: sender signs hash of message with their private key → recipient verifies with sender's public key
