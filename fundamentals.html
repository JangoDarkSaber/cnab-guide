Two's Complement (Signed Integers)

Representing Negative Numbers

Two's complement is the standard representation for signed integers on virtually all modern processors. To negate a number: (1) invert all bits (one's complement), then (2) add 1.

# Represent -5 in 8-bit two's complement Step 1: +5 in binary = 0000 0101 Step 2: Invert bits = 1111 1010 (one's complement) Step 3: Add 1 = 1111 1011 = -5 # Key property: MSB=1 means negative in signed interpretation # 0xFF = 255 (unsigned) OR -1 (signed 8-bit) # 0x80 = 128 (unsigned) OR -128 (signed 8-bit) # 0x7F = 127 = max positive signed 8-bit value

Bit Masking & Bitwise Operations

# AND (&) — keeps only bits that are 1 in BOTH operands 1010 1100 (0xAC = 172) & 1111 0000 (0xF0 = mask: keep upper nibble) = 1010 0000 (0xA0 = 160) ← IP subnet masking works this way # OR (|) — sets bit if 1 in EITHER operand 0100 0000 | 0000 0001 = 0100 0001 ← set multiple flags at once # XOR (^) — 1 only if bits DIFFER 1100 XOR 1010 = 0110 ← XOR same value twice = original (encryption) # Left shift (<<) = multiply by 2^n 0000 0001 << 3 = 0000 1000 (1 → 8) # Right shift (>>) = divide by 2^n 0001 0000 >> 2 = 0000 0100 (16 → 4)
// Exam Tip: Subnet Masking is Bitwise AND
192.168.1.74 AND 255.255.255.0 = 192.168.1.0 (the network address). ANDing an IP with its subnet mask zeros out the host portion, revealing the network portion. This is how routers determine if a destination is local or remote.

CPU Architecture
CISC · RISC · x86 · ARM · registers · instruction cycle · cache

ISA Families

CISC — Complex Instruction Set Computer
  • Philosophy: Single instructions can do complex multi-step operations (load + compute + store)
  • Examples: x86, x86-64 (AMD64), IBM mainframe
  • Characteristics: Variable-length instructions, many addressing modes, microcode translates to micro-ops internally
  • Trade-off: Complex decoder hardware; powerful individual instructions
  • Use case: Desktops, servers, laptops — general-purpose computing
RISC — Reduced Instruction Set Computer
  • Philosophy: Simple, uniform instructions executed in one clock cycle
  • Examples: ARM (mobile/IoT), MIPS (routers/embedded), RISC-V (open), PowerPC
  • Characteristics: Fixed-length instructions, load/store architecture (only load/store access memory), more general-purpose registers
  • Trade-off: More instructions needed per task; simpler, faster decoder
  • Use case: Mobile (ARM in every smartphone), embedded systems, IoT devices

Fetch-Decode-Execute Cycle

The Instruction Cycle
  • Fetch: CPU reads the instruction at the address in RIP/EIP from memory into the Instruction Register (IR). RIP increments to the next instruction.
  • Decode: Control unit decodes the instruction — determines opcode, operands, addressing mode
  • Execute: ALU (Arithmetic Logic Unit) performs the operation. May involve reading/writing registers or memory (memory access stage)
  • Write-back: Result written to destination register or memory location. Flags updated.
  • Modern CPUs pipeline these stages — multiple instructions in flight simultaneously, out-of-order execution

Endianness
big-endian · little-endian · network byte order · why it matters

Definition

What Is Endianness?

Endianness describes the byte order in which multi-byte values are stored in memory. For the 32-bit value 0x0A0B0C0D, the question is which byte goes at the lowest memory address. The two main orderings are big-endian (most significant byte first) and little-endian (least significant byte first).

Why It Matters

  • Network byte order is big-endian. Network stack functions like htons() (host to network short) and htonl() convert between host byte order and network order. On x86 (LE), this swaps bytes; on BE systems, it's a no-op.
  • Packet analysis: In Wireshark/tcpdump, multi-byte fields in IP/TCP headers are displayed in big-endian (as-transmitted) order. If you read a raw hex dump of a packet, multi-byte values appear MSB-first.
  • Exploit development: When writing 32/64-bit addresses in shellcode targeting x86, addresses must be written in little-endian (LSB first). 0xDEADBEEF = \xEF\xBE\xAD\xDE in memory.
  • File format parsing: ELF binaries specify endianness in the header (EI_DATA field: 1=LE, 2=BE). PE (Windows) is always little-endian.
// Bi-Endian Systems
ARM, MIPS, and PowerPC support both endianness modes via a configuration bit. They're called bi-endian. In practice, ARM is usually configured as little-endian in modern devices. MIPS-based routers are often big-endian, which matters when writing exploits for router firmware.

Virtualization
hypervisors · type 1 vs type 2 · containers · VMs · security

Virtualization Types

Type 1 Hypervisor (Bare Metal)

Runs directly on hardware — no host OS underneath. The hypervisor IS the OS for the hardware. VMs run on top of it. Provides best performance and isolation since there's no host OS overhead.

  • VMware ESXi / vSphere
  • Microsoft Hyper-V (when run as Windows role on bare metal)
  • Xen (used by AWS EC2)
  • KVM (Linux kernel built-in; technically hybrid)
Type 2 Hypervisor (Hosted)

Runs as an application on top of a host OS. Host OS manages hardware; hypervisor manages guest VMs as processes. More overhead but easier to set up on existing machines.

  • VMware Workstation / Fusion
  • Oracle VirtualBox
  • Parallels Desktop (macOS)
  • QEMU (user mode)

CPU Virtualization Extensions

Intel VT-x / AMD-V

Hardware-assisted CPU virtualization. Adds VMX (Virtual Machine Extensions) instructions. Enables a VMX root mode (Ring -1) for the hypervisor that is more privileged than Ring 0. Guest OSes run in non-root mode (Ring 0 for guest kernel, Ring 3 for guest apps). Without VT-x, hypervisors needed software-based binary translation (slow). Required to run 64-bit guests and nested virtualization.

SLAT (Second Level Address Translation)

Intel EPT (Extended Page Tables) / AMD RVI (Rapid Virtualization Indexing). Hardware-assisted memory virtualization. Maps guest physical addresses → host physical addresses in hardware, without hypervisor intervention on every memory access. Dramatically reduces VM memory overhead. Required for Hyper-V core functionality in Windows.

// VM Detection (Security Relevance)
Malware often detects virtualized environments to avoid analysis. Detection methods: check for VMware registry keys, CPUID hypervisor bit (leaf 0x1, ECX bit 31), timing differences (RDTSC instruction), presence of VM guest additions, MAC address prefixes (VMware=00:0C:29, VBox=08:00:27), and small physical RAM size.

Stack Frame Anatomy

How the Stack Works

The stack is a LIFO (Last In, First Out) region used for function calls. The stack pointer (RSP) always points to the top (lowest address on x86). When a function is called:

# Call convention (System V AMD64 ABI — Linux) 1. Arguments 1-6 in RDI, RSI, RDX, RCX, R8, R9 2. CALL pushes return address onto stack (RSP -= 8) 3. Callee pushes RBP, sets RBP = RSP (prologue) 4. Local variables allocated by RSP -= N # Stack frame layout (top = lower addresses) RSP → [local variables] ← grows here [saved RBP] ← frame pointer RBP → [return address] ← where to go after RET [caller's frame] # Stack overflow: overwrite return address → control flow hijack

Offsets and Lengths

Working with Memory Offsets

An offset is a byte count from a base address to locate a field or element. Used everywhere in protocols, file formats, and binary exploitation.

# Example: IPv4 header offsets Offset 0: Version + IHL (1 byte) Offset 1: DSCP + ECN (1 byte) Offset 2: Total Length (2 bytes) Offset 4: Identification (2 bytes) Offset 6: Flags + Frag Off (2 bytes) Offset 8: TTL (1 byte) Offset 9: Protocol (1 byte) ← 6=TCP 17=UDP 1=ICMP Offset 10: Header Checksum (2 bytes) Offset 12: Source IP (4 bytes) Offset 16: Destination IP (4 bytes) # Address calculation: base_addr + offset = field_addr # struct foo { int a; char b[4]; int c; } # c is at offset 8 (4 bytes for a + 4 bytes for b)

Paging & Virtual Memory

  • Page: Fixed-size block of virtual memory, typically 4 KB (4096 bytes). Memory is allocated in page units.
  • Page Table: OS-maintained mapping from virtual page numbers to physical page frames. CPU's MMU (Memory Management Unit) uses it for every memory access.
  • TLB (Translation Lookaside Buffer): Cache of recent virtual→physical translations. A TLB miss forces a costly page table walk.
  • Page Fault: CPU raises exception when accessing an unmapped or protected page. OS handles it: map more memory, load from swap, or kill process (segfault).
  • ASLR (Address Space Layout Randomization): Security mitigation — randomizes base addresses of stack, heap, and libraries at each run, making buffer overflow exploitation harder.
  • DEP/NX (Data Execution Prevention / No-Execute): CPU bit marking pages as non-executable. Stack and heap pages marked NX prevent shellcode execution on the stack.

// SSD TRIM & Forensics
When you delete a file on an SSD with TRIM enabled, the OS sends a TRIM command telling the SSD controller those blocks are free — the controller may immediately zero them as part of garbage collection. Unlike HDDs where deleted data persists until overwritten, SSD TRIM makes file recovery much harder or impossible. This is critical knowledge for digital forensics — image SSD evidence IMMEDIATELY; every passing second may destroy deleted file artifacts.

Kernel Space vs User Space
privilege rings · syscalls · context switching · security boundary

The Fundamental Separation

Kernel Space (Ring 0)
  • Highest privilege — direct hardware access
  • OS kernel, device drivers, kernel modules run here
  • Can execute any CPU instruction (including privileged ones: IN, OUT, HLT, LGDT)
  • Can access any physical memory address
  • A bug here can crash or compromise the entire system
  • Context: kernel tasks, interrupt handlers, system call handlers
User Space (Ring 3)
  • Restricted privilege — cannot directly access hardware
  • All user applications, daemons, and libraries run here
  • Cannot execute privileged CPU instructions
  • Can only access its own virtual memory pages (MMU enforced)
  • A crash only affects the process, not the OS
  • Must request kernel services via system calls

System Call Interface

How User Space Talks to the Kernel

When a user-space program needs a kernel service (open a file, create a socket, allocate memory), it executes a system call. The CPU switches privilege level from Ring 3 → Ring 0, the kernel handles the request, and control returns to user space.

# Linux system call mechanism (x86-64) 1. Put syscall number in RAX (e.g., 0=read 1=write 2=open) 2. Put arguments in RDI, RSI, RDX, R10, R8, R9 3. Execute SYSCALL instruction → Ring 0 4. Kernel validates args, performs operation, returns result in RAX 5. SYSRET → back to Ring 3 # Common Linux syscall numbers 0 = read 1 = write 2 = open 3 = close 57 = fork 59 = execve 60 = exit 41 = socket 42 = connect

Context Switching

What Happens During a Context Switch

When the OS scheduler switches from one process to another (or from user to kernel), a context switch occurs: the current CPU state (all registers, instruction pointer, flags, memory mappings) is saved to the process's PCB (Process Control Block), and the next process's saved state is loaded. Context switches are expensive (~1–10 µs) because they may invalidate caches (TLB flush) and require saving/restoring many registers. High context-switch rates can seriously degrade performance.

Security Implications of the Boundary

  • Privilege escalation attacks target vulnerabilities that allow a user-space process to gain kernel-level (Ring 0) execution — e.g., kernel exploits, driver bugs
  • Sandbox escapes: browsers run renderer processes in restricted user-space sandboxes; escaping the sandbox then escaping user space entirely requires chaining two vulnerabilities
  • Rootkits: Kernel-mode rootkits live in Ring 0 and can hide files/processes from user-space security tools by patching kernel data structures or system call tables
  • seccomp (Linux): Restricts which system calls a process can make — reduces kernel attack surface. Used by Docker, Chrome, ssh
  • SMEP/SMAP (Intel): Supervisor Mode Execution/Access Prevention — prevents kernel code from executing user-space pages or accidentally dereferencing user-space pointers

RISC vs CISC Key Distinctions

RISC Characteristics
  • Instructions start with R then a number (e.g., R1, R2) — written in assembly language
  • One register = one action (load/store architecture)
  • Fixed-length instructions; simple decoder
  • More general-purpose registers; arithmetic only on registers
  • Examples: ARM, MIPS, SPARC, RISC-V, PowerPC
CISC Characteristics
  • 3-letter register names used to direct the register (e.g., EAX, EBX, RDI)
  • Single instruction can load + compute + store
  • Variable-length instructions; complex decoder
  • Microcode translates complex instructions internally
  • Examples: x86, x86-64 (Intel/AMD)

32-bit vs 64-bit Architecture

32-bit (x86)
  • Processes up to 4 bytes of data per instruction
  • Maximum addressable RAM: 4 GB (2³² addresses)
  • Registers: EAX, EBX, ECX, EDX, ESI, EDI, ESP, EBP, EIP
  • Pointer size: 4 bytes
  • OS label: x86, i386, i686
64-bit (x86-64)
  • Processes up to 8 bytes of data per instruction
  • Addressable RAM: theoretically 16 EB (2⁶⁴); practical hardware limit much lower
  • Extended registers: RAX, RBX, RCX, RDX + R8–R15
  • Pointer size: 8 bytes
  • OS label: x86_64, amd64

Compiled vs Interpreted Languages
compilers · interpreters · runtime · error detection · examples

Compiled Languages

How Compilation Works

A compiler takes the whole program as input and translates it to an executable binary in several steps (preprocessing → parsing → optimization → code generation → linking). The resulting binary is platform-specific — it depends on the CPU architecture and OS.

  • Compilation happens once; the binary runs as many times as needed without the compiler
  • Binary is not portable — compiled for a specific architecture (x86-64 binary won't run on ARM)
  • Compilers catch syntax and type errors before the program ever runs — compilation fails if present
  • Generally faster execution since translation to machine code already done
  • Example: C, C++, Go, Rust, Swift
# C compile and run gcc -o program program.c # Compile once → binary ./program # Run directly, no compiler needed

Interpreted Languages

How Interpretation Works

An interpreter reads and executes the program instruction by instruction at runtime. Each instruction is translated to machine code and run immediately. No standalone binary is produced.

  • The interpreter must be present in RAM every time the program runs
  • Source code is portable — run on any system with the interpreter installed
  • Errors are caught at runtime — a bug in a rarely-executed branch may not surface until production
  • Generally slower than compiled code (overhead of runtime translation)
  • Example: Python, Ruby, JavaScript, Perl, PHP, Bash
# Python — interpreter required at runtime python3 program.py # Interpreter reads + executes line by line

Java compiles source to bytecode (.class files) — not native machine code. The JVM (Java Virtual Machine) interprets or JIT-compiles bytecode at runtime. Stack vs Heap Memory
allocation · lifetime · static vs dynamic · swap · pagefile

Stack vs Heap

Stack
  • Instructions and local variables for a program are statically allocated by the program at compile time
  • LIFO (Last In, First Out) — grows downward on x86
  • Managed automatically by the compiler/CPU — push on call, pop on return
  • Fixed max size (stack overflow if exceeded)
  • Fast allocation: just decrement the stack pointer
  • Lifetime: exists only for the duration of the function call
Heap
  • Instructions are dynamically assigned from the program at runtime (malloc/new)
  • Data is usually discarded once it has completed its use (free/delete)
  • Managed by the programmer (or garbage collector in managed languages)
  • Grows upward; size limited only by available memory
  • Slower allocation (memory allocator must find free block)
  • Lifetime: persists until explicitly freed — memory leaks occur if not freed

Unix Swap File

Virtual Memory via Swap

A swap file (or swap partition) allows the OS to use hard disk space to simulate extra memory. When RAM is full, the OS moves least-recently-used pages to disk. When that program is accessed again, it swaps back into RAM. This enables running more programs than would fit in RAM alone — but causes significant slowdown due to disk I/O speed being far below RAM speed. RAM + swap = virtual memory.

Windows Page File (pagefile.sys)

Demand Paging via pagefile.sys

Windows implements virtual memory via demand paging using pagefile.sys. The MMU (Memory Management Unit) on the CPU manages memory in page frames. When RAM fills, Windows identifies least recently used page frames and writes them to pagefile.sys on disk, freeing RAM for active processes. When the paged-out program is needed again, Windows swaps it back. This is handled transparently by the Memory Manager.

// Security Note
pagefile.sys may contain sensitive data (encryption keys, passwords, credentials) from programs that have run. On decommissioned systems, it should be securely wiped.

// Forensic Significance
The volatile/non-volatile distinction is critical in digital forensics. The order of volatility dictates evidence collection order: registers/cache (lost immediately on power-off) → RAM (lost on power-off) → swap/pagefile → disk. Always capture volatile memory first.

// Why drives "appear" smaller in Windows
A "1 TB" drive has 10¹² bytes (manufacturer, SI). Windows reports in GiB (÷1024³) = ~931 GiB, which it shows as "931 GB". The drive isn't smaller — just different units.
