Shell Expansion Types (in order of evaluation)

  • Brace expansion: cp file{.bak,.orig} → cp file.bak file.orig. Not POSIX, bash-specific.
  • Tilde expansion: ~ → $HOME, ~user → that user's home, ~- → previous directory ($OLDPWD)
  • Parameter expansion: ${VAR}, with modifiers like ${VAR:-default}
  • Command substitution: $(cmd) or `cmd` — output replaces expression
  • Arithmetic expansion: $((expr)) — integer arithmetic only
  • Word splitting: unquoted expansions split on IFS characters (space/tab/newline). Always quote "$VAR" to prevent.
  • Pathname (glob) expansion: *=any chars, ?=one char, [abc]=character class, [!abc]=negation
  • Process substitution: <(cmd) and >(cmd) — treat command I/O as a file (bash/zsh only)

Hard Links vs Symbolic Links

Hard Links

Another directory entry pointing to the same inode (same physical data). Deleting the original doesn't remove data until all hard links are gone (link count = 0). Cannot cross filesystems. Cannot link directories. ls -li shows same inode number.

ln file hardlink # Creates hard link
Symbolic (Soft) Links

A special file containing a path string pointing to another file/dir. Can cross filesystems and link to directories. If target is deleted, symlink becomes a dangling link. Symlinks have their own inode but no independent data.

ln -s /target /link # Creates symlinkreadlink -f link # Resolve to absolute path

NTFS-Specific Features

Alternate Data Streams (ADS)

NTFS allows multiple data streams per file. Main content is in the default stream (file.txt); additional streams are file.txt:streamname. Used legitimately (Zone.Identifier marks downloaded files) and by malware to hide data. Normal dir doesn't show ADS.

echo hidden > file.txt:secret dir /r # Shows ADS streams more < file.txt:secret
NTFS Permissions (ACLs)

Fine-grained ACLs per file/folder. Permissions: Full Control, Modify, Read & Execute, Read, Write, List Folder Contents. Explicit Deny overrides Allow. Effective permissions = union of all group memberships.

icacls C:\folder # Show ACL icacls file /grant User:(R) icacls file /deny User:(W) cacls # Legacy; use icacls instead

Hashing — Deep Dive

Properties of Cryptographic Hash Functions
  • Deterministic: Same input always produces same output
  • One-way (preimage resistance): Cannot reverse hash → original
  • Second preimage resistance: Given input X, cannot find different Y with same hash
  • Collision resistance: Cannot find ANY two inputs with same hash
  • Avalanche effect: Single bit change → completely different hash (~50% of bits flip)
  • Fixed output length: Always same size regardless of input size

Password Hashing: Salt vs Unsalted

Unsalted Hash (Weak)

Same password always produces same hash. Enables rainbow table attacks — precomputed hash→password lookup tables. Attacker can identify users with same password (identical hashes in /etc/shadow).

MD5("password") = 5f4dcc3b5aa765d61d...# Always the same — look it up in rainbow table
Salted Hash (Strong)

A random per-user salt is appended/prepended before hashing. Salt stored in plaintext alongside hash. Makes rainbow tables useless — each user needs unique attack. Makes identical passwords produce different hashes.

SHA256("password" + "xK7mQ9") = 3a8f...SHA256("password" + "Zp2nW1") = 91c4...# Same password, different salt = different hash

Malware Evasion Techniques

Evading Signature Detection
  • Packing: Compress/encrypt binary; signature not visible. Unpacks at runtime. UPX is common example.
  • Polymorphism: Code mutates on each infection while preserving function. Signature becomes useless.
  • Metamorphism: Code completely rewrites itself — equivalent instructions, different bytes every time.
  • Encryption: Payload encrypted; decryptor stub may be detectable but payload isn't scanned.
  • Obfuscation: Insert NOPs, junk code, rename variables to change byte patterns.
  • Living off the Land (LotL): Use legitimate OS tools (powershell, wmic, regsvr32) — these won't match malware signatures.
Evading Behavioral / Sandbox
  • Sandbox detection: Check for VM artifacts (VMware registry keys, CPUID, small RAM), sleep for days before executing, check user mouse movement.
  • Timing attacks: NtDelayExecution with huge sleep — many sandboxes time out and classify as benign.
  • Environment checks: Only execute if specific username, domain, or software is present.
  • Process injection: Inject into trusted process (explorer.exe, svchost.exe) so behavior attributed to legitimate process.
  • Fileless malware: Execute entirely in memory, never write to disk — evades file-based scanning.

Types of Malware

Virus

Attaches to and modifies host files. Requires user action to spread. Infects executable files or boot sectors.

Worm

Self-replicating; spreads without user interaction via network vulnerabilities or email. Can cause network saturation.

Trojan

Disguised as legitimate software. No self-replication. User installs it willingly. Often delivers backdoor/RAT.

Rootkit

Hides itself and other malware. Kernel-level rootkits patch OS to hide files, processes, network connections from normal tools.

Ransomware

Encrypts victim's files; demands payment for decryption key. Double extortion: also exfiltrates data before encrypting.

Spyware / Keylogger

Silently records keystrokes, screenshots, credentials. Exfiltrates to attacker. Often bundled with freeware.

File Integrity Monitoring (FIM) Concepts

What FIM Does

File Integrity Monitoring tools create a cryptographic baseline database capturing the hash, size, permissions, ownership, and timestamps of monitored files. Periodic or continuous scans compare current state to the baseline and alert on any unauthorized changes. FIM is essential for detecting rootkits, backdoors, unauthorized configuration changes, and fulfilling compliance requirements (PCI-DSS Requirement 11.5, HIPAA, SOX).

Tripwire (Unix)

How Tripwire Works
  • Configuration defines which files/dirs to monitor and what properties to check
  • --init creates the initial baseline database (stored encrypted)
  • --check compares current filesystem to baseline; reports added/removed/modified files
  • Policy file uses rules to define severity levels for different change types
  • Database and policy files are signed with a site key and local key to prevent tampering
  • After approved changes (patches, updates), --update refreshes baseline
Tripwire Commands
twadmin --create-polfile twpol.txt # Compile policy tripwire --init # Initialize database tripwire --check # Run integrity check tripwire --update # Update after approved changes tripwire --check --email-report # Email violations

AIDE (Advanced Intrusion Detection Environment)

AIDE — Open Source Alternative to Tripwire

Commonly used on Linux. Configured via /etc/aide/aide.conf or /etc/aide.conf. Monitors file attributes selectable per rule: p=permissions, u=uid, g=gid, s=size, m=mtime, c=ctime, sha256=content hash. Typical deployment: run aide --init after clean install, then schedule daily aide --check.

aide --init # Create baseline db mv /var/lib/aide/aide.db.new /var/lib/aide/aide.db aide --check # Compare current state to baseline aide --update # Update baseline after approved change

Windows File Protection / Windows Resource Protection

WFP / WRP Mechanism
  • Introduced in Windows Me/2000 as Windows File Protection (WFP); renamed Windows Resource Protection (WRP) in Vista
  • Protects ~3,000 critical system files and registry keys via ACLs — only TrustedInstaller can modify them
  • If a protected file is replaced, WRP restores the correct version from the Windows Side-by-Side (WinSxS) component store automatically
  • Restoration happens silently in the background without user intervention
  • Prevents rootkits and malware from replacing core system DLLs (DLL hijacking protection)

Command-line tool to manually scan and repair WRP-protected files. Results logged to C:\Windows\Logs\CBS\CBS.log.

// FIM vs Antivirus
AV detects malware at execution time by matching known signatures/behaviors. FIM detects that something changed on disk — useful for detecting post-compromise modifications to config files, scripts, and binaries that AV might miss. They complement each other.

How Code Signing Works

The Full Process
  • Developer obtains a code-signing certificate from a trusted Certificate Authority (DigiCert, Sectigo, etc.). CA verifies the identity of the requester (OV or EV validation).
  • Hash is computed over the binary/script (e.g., SHA-256 of the PE file).
  • Hash is encrypted with developer's private key → this produces the digital signature.
  • Signature + certificate chain embedded in the file (PE Authenticode = WIN_CERTIFICATE structure appended to binary).
  • Timestamping: Signature is countersigned by a Timestamp Authority (TSA) so it remains valid even after the signing certificate expires.
  • Verification at execution: OS decrypts signature with the public key from the certificate, recomputes the hash of the binary, and compares. If they match AND the certificate chains to a trusted root AND isn't revoked → valid.

Certificate Types for Signing

TypeValidation LevelSmartScreen Reputation
DV (Domain Validated)Proves domain control only. Not typical for code signing.Weak — minimal trust

EV (Extended Validation)Most rigorous: in-person identity verification, phone calls, legal filings checkImmediate SmartScreen trust — no warning on first run

Certificate Revocation

CRL (Certificate Revocation List)

CA periodically publishes a signed list of revoked certificate serial numbers. Client downloads CRL and checks if certificate is listed. Problem: CRLs can be large, are updated periodically (not real-time), and require internet access. A revoked cert may still be trusted if CRL isn't current.

OCSP (Online Certificate Status Protocol)

Client queries CA's OCSP responder with certificate serial number and gets real-time good/revoked/unknown response. Faster and more current than CRL. OCSP Stapling: server pre-fetches its own OCSP response and serves it to clients — improves speed, reduces privacy leaks to CA.

The NT 5 Family — XP / Server 2003

NT 5.x Security Profile
  • No ASLR — all binaries load at predictable addresses. Exploits are highly reliable and portable across all XP machines.
  • No UAC — users run as full administrator by default. Any code that executes gets admin rights without prompting.
  • DEP added late — only with XP SP2 (2004) and only for data execution. Easily bypassed via return-to-libc.
  • No driver signing requirement — any unsigned driver loads silently in 32-bit XP.
  • LM hash storage — older authentication; LM hashes trivially crackable (7-char split, uppercase only).
  • Dominant attack era: 2001–2014. MS08-067 (Conficker), MS03-026 (Blaster), MS04-011 (Sasser).

Why Kernel Version Matters for Security Work

Exploit Reliability by NT Version
  • NT 5.x target: No ASLR, no UAC, predictable addresses. Buffer overflow → reliable shellcode execution. Exploits are portable.
  • NT 6.x target: Need info-leak to defeat ASLR. Need UAC bypass or high-priv context. Exploit chains are longer.
  • NT 10 + VBS: Credential Guard defeats in-memory credential theft. HVCI defeats most kernel rootkits. Exploit chains require hypervisor-level bugs.
  • Identifying target OS version is one of the first steps in a penetration test — it determines which techniques are viable.

Device Drivers

CPU Protection Rings

Privilege Levels

Modern CPUs implement privilege rings: Ring 0 (Kernel mode) has unrestricted access to all hardware and memory — the OS kernel and device drivers run here. Ring 3 (User mode) is where applications run — restricted; cannot directly access hardware or arbitrary memory. Rings 1 and 2 are rarely used on x86 (hypervisors sometimes use Ring -1 / VMX root mode). A kernel-mode driver bug or compromise = full system compromise. This is why driver signing is enforced on 64-bit Windows.

Windows Drivers
  • Location: C:\Windows\System32\drivers\ for kernel drivers (.sys)
  • Registry: HKLM\SYSTEM\CurrentControlSet\Services — each service/driver has ImagePath, Type, Start entries
  • Type values: 1=kernel driver, 2=file system driver, 4=adapter, 8=recognizer
  • 64-bit Windows: Requires kernel-mode drivers to be digitally signed (WHQL or cross-signed). Unsigned drivers blocked by default (Secure Boot enforced on modern systems)
  • KMCS: Kernel Mode Code Signing — part of Windows integrity policy
  • DSE bypass: Attackers use vulnerable signed drivers (BYOVD — Bring Your Own Vulnerable Driver) to disable DSE and load unsigned malicious drivers
Linux Kernel Modules
  • Location: /lib/modules/$(uname -r)/ — organized by type
  • Extension: .ko (kernel object)
  • Loading: modprobe (resolves deps) or insmod (raw load)
  • Unloading: rmmod module_name (fails if in use)
  • Config: /etc/modules for modules to load at boot; /etc/modprobe.d/ for options/blacklists
  • Module signing: Linux supports module signing via CONFIG_MODULE_SIG. Required on UEFI Secure Boot systems

Hard Drive Partitioning & File Systems

Partition Table Types

MBR (Master Boot Record)
  • Located in first 512-byte sector (LBA 0) of disk
  • Contains: 446-byte bootstrap code + 64-byte partition table (4 entries × 16 bytes) + 2-byte signature (0x55AA)
  • Max 4 primary partitions. Workaround: use 1 extended partition containing logical drives
  • Max partition/disk size: 2TB (32-bit LBA addressing)
  • Used with BIOS firmware. Legacy boot method.
  • Single point of failure — damage to first sector = unbootable
GPT (GUID Partition Table)
  • Part of UEFI standard. Replaces MBR limitations.
  • Supports up to 128 partitions (no extended partition workaround)
  • Max partition size: 9.4 ZB (zettabytes — effectively unlimited)
  • Header and partition table replicated at both start AND end of disk (redundancy)
  • Each partition has a 128-bit GUID. Protective MBR preserves backward compatibility.
  • Required for UEFI/Secure Boot on modern systems

Journaling Explained

Why Journaling Matters

Without journaling: if power fails during a write, the filesystem can be left in an inconsistent state requiring slow full fsck scan. A journal (write-ahead log) records intended operations before committing. On recovery, the journal is replayed or rolled back — consistency restored in seconds. Modes: journal = log data+metadata (safest, slowest); ordered = data written first, then metadata logged (default ext4); writeback = only metadata logged (fastest, data may be stale on crash).

Inodes (Unix)

What Is an Inode?

Every file/directory on a Unix filesystem is represented by an inode — a data structure storing metadata: file type, permissions (mode), owner UID/GID, file size, timestamps (atime/mtime/ctime), link count, and pointers to data blocks. The inode does NOT store the filename — filenames are stored in directory entries that map names to inode numbers. A filesystem has a fixed inode table created at format time — you can run out of inodes before running out of disk space (common with millions of small files).

ls -i file # Show inode number stat file # Full inode details df -i # Show inode usage (not space usage) tune2fs -l /dev/sda1 | grep inode # Filesystem inode info

WOW64 Emulator

How 32-bit Apps Run on 64-bit Windows

WOW64 (Windows 32-bit On Windows 64-bit) is the x86 emulator subsystem that allows 32-bit applications to run on 64-bit Windows. When a 32-bit application tries to access C:\Windows\System32, WOW64 transparently redirects it to SysWOW64 — so the 32-bit app gets the 32-bit DLLs it needs without modification. The 64-bit System32 folder is untouched.

DLL vs Linux Shared Object

Windows DLL ↔ Linux .so

Dynamically linked libraries (Windows .dll) and shared objects (Linux .so) are conceptually the same thing. Both are containers for executable code and data that can be loaded into a process at runtime, shared across multiple processes, and updated independently of the programs that use them. Key difference: Windows uses the PE format for DLLs; Linux uses the ELF format for .so files.

DEP (Data Execution Prevention)

DEP — Memory Page Protection

DEP is a system-level memory protection feature built into Windows starting with Windows XP and Windows Server 2003. DEP enables the system to mark one or more pages of memory as non-executable — code cannot run in regions marked as data (stack, heap). Implemented via the CPU's NX bit (Intel) / XD bit (AMD). Prevents shellcode injection attacks that rely on executing data placed in non-code memory regions.

Hard Drive Physical Structure

ComponentDescription
PlattersPhysical magnetic disks inside the HDD. Each platter provides top and bottom recording surfaces. A drive may have one or several platters, each with its own read/write head pair.
TracksConcentric circular rings on a platter surface. Data is organized into tracks.
SectorsSubdivision of a track. Each sector stores a fixed amount of user-accessible data. The sector is the minimum storage unit of a hard drive — traditionally 512 bytes (Advanced Format drives: 4096 bytes).
PartitionsLogical sections of the hard drive separated from other segments. Each partition can have its own filesystem.
InodesProvide important information on files: user/group ownership, access mode (R-W-X permissions), and file type. Each file has exactly one inode.
LogicalRefers to the virtual space within the disk — partitions, filesystems, logical volumes.
PhysicalRefers to the actual hardware — platters, heads, sectors, the physical device itself.

Inode Contents (Full Detail)

What an Inode Stores
  • File type — regular file, directory, symlink, device, socket, pipe
  • File size in bytes
  • Timestamps — creation, access (atime), modification (mtime), metadata change (ctime)
  • File permissions — read, write, execute for owner/group/other
  • Access Control List — extended permissions for specific users/groups
  • File protection flags
  • Link count — number of hard links pointing to this inode
  • File pointers — addresses of the storage blocks containing file data
  • UID/GID (owner and group)
What an Inode Does NOT Store
  • Filename — stored in directory entries that map name → inode number
  • Actual file data — stored in data blocks pointed to by the inode
// Why this matters
Separating name from inode enables: renaming a file without affecting its inode number; multiple filenames (hard links) pointing to the same data; directories treated as files with their own inodes.

Hard Links vs Symbolic Links (CNAB Reference)

Symbolic (Soft) Link

Like a Windows shortcut. The file/directory is at one point on disk; the link is a pointer to it. Each symlink has its own inode number, allowing symlinks between different filesystems.

ln -s test link-to-test # Create symlink # link-to-test has different inode than test # Deleting test = dangling link
Hard Link

Associates two or more filenames sharing the same inode. Each hard link is an exact copy in terms of data and permissions. Changes to one propagate to all. Hard links cannot span filesystems or link directories.

ln test link-hard-test # Create hard link ls -li # Both show same inode number # 73793 test 73793 link-hard-test

Boot Processes & Linux Runlevels
Windows 10 boot · Unix boot · GRUB · init · runlevels · BOOTP/DHCP

Windows 10 Boot Sequence

4-Stage Process
  1. Pre-Boot — Power-on Self Test (POST). Loads firmware settings. Checks for valid disk system and valid MBR.
  2. Windows Boot Manager — Checks if multiple OSes on disk. Starts winload.exe.
  3. Windows OS Loader — winload.exe starts important processes to kick-start the Windows kernel.
  4. Windows NT OS Kernel — Picks up registry settings and additional drivers. Control taken by the Session Manager process (smss.exe). GUI is loaded.

Unix/Linux Boot Sequence

6-Stage Process
  1. BIOS — Executes the MBR (Master Boot Record)
  2. MBR — Executes GRUB (Grand Unified Boot Loader)
  3. GRUB — Executes the Kernel
  4. Kernel — Executes /sbin/init (PID 1)
  5. Init — Executes runlevel programs
  6. Runlevel — Programs executed from /etc/rc.d/rc*.d/ (Runlevel 3 is default for servers)

Linux Runlevels

Defined in /etc/inittab which contains descriptions of the different run states the system can assume.

RunlevelDescription
0Halt the system
1Single-user mode (for special administration — no network, maintenance)
2Local Multiuser with Networking but without network services (like NFS)
3Full Multiuser with Networking (default on most servers)
4Not Used (reserved / custom)
5Full Multiuser with Networking and X Windows GUI (X11) — default on desktops
6Reboot
// systemd Targets
Modern Linux with systemd uses targets instead of runlevels: multi-user.target ≈ RL3, graphical.target ≈ RL5, rescue.target ≈ RL1, poweroff.target ≈ RL0, reboot.target ≈ RL6.

BOOTP vs DHCP

BOOTP

Allows a diskless client machine to discover its own IP address, the address of a server host, and the name of a file to be loaded into memory and executed. Static assignments — no lease concept.

DHCP

Based on BOOTP; adds automatic allocation of reusable network addresses and additional configuration options. DHCP can interoperate with BOOTP clients via relay agents. Process: Discover → Offer → Request → Acknowledge.

DNS Resolver Configuration

Linux: /etc/resolv.conf

Defines how the system uses DNS to resolve hostnames and IPs. Contains up to three nameserver lines and search domain configuration.

# /etc/resolv.conf example search us.mydomain.com mydomain.com nameserver 192.168.154.3 nameserver 192.168.154.4 nameserver 10.216.106.3 # DHCP typically writes this file automatically
Windows: hosts file

Plain text file at C:\Windows\System32\drivers\etc\hosts. Maps hostnames to IP addresses. Checked before DNS. Owned by NT AUTHORITY\SYSTEM — only modifiable by administrator. Attacker modification = DNS hijacking on the local machine.

Tasklist, TTL & Windows SIDs
TASKLIST /SVC · TTL values · SID table · hardware/software/firmware

TASKLIST — Querying Services and PIDs

Query by Service Name

Find the PID of a process hosting a specific service. Example: find PID of the Print Spooler service:

TASKLIST /SVC /FI "SERVICES eq Spooler" # Output shows PID in the central column
Query by PID

Before force-killing a process, confirm what services would be stopped. Example: see all services running in PID 1284:

TASKLIST /SVC /FI "PID eq 1284" # May show 5+ services in one svchost
Remote Machine Query Flags

TASKLIST can interrogate remote machines. Add /S (system), /U (username), and /P (password) flags:

TASKLIST /S \\RemotePC /U domain\user /P password /SVC

Bit Bucket

/dev/null — The Bit Bucket

The bit bucket is where lost or discarded computer data has gone. Any data which does not end up where it is supposed to — lost in transmission, from a crash, or explicitly discarded — is said to have gone to the bit bucket.

# Unix: /dev/null is the bit bucket cmd > /dev/null 2>&1 # Discard all output # Windows: NUL is the bit bucket cmd > NUL # Discard stdout
